The stack decision

Why BEAM.
Why Elixir.
Why not Java.

Java works. Go works. Python works. This is not an argument that they don't. It's an argument that for the specific problem Foundry solves — governed, long-lived, regulated domain platforms — one stack is structurally better, and the gap compounds over time.

01 — The runtime

BEAM was built for systems
that cannot fail.

Ericsson designed the BEAM virtual machine for telephone exchanges in 1986. The design constraint was unforgiving: millions of concurrent connections, hardware failures, software crashes, zero tolerance for cascading failure. Those constraints produced a runtime with properties no other mainstream VM shares.

01 — Concurrency

Two kilobytes.
Millions of processes.

Every concurrent operation runs in its own isolated process — 2KB initial heap, preemptively scheduled, garbage collected independently. A crash in one process cannot corrupt another. You don't design around this. It's the execution model. No thread pools to tune. No async/await ceremony. Concurrency is the default, not a feature you add.

100k concurrent users · same hardware
02 — Fault isolation

Supervisors restart.
Systems continue.

OTP supervision trees arrange processes in hierarchies. When a process crashes, its supervisor restarts it according to a declared strategy — one_for_one, one_for_all, rest_for_one. The rest of the system never knew. This isn't error handling. It's fault architecture. The difference between catching exceptions and designing so that exceptions don't matter.

nine nines availability · Ericsson AXD301
03 — Consistent latency

Preemptive scheduling.
No GC pauses.

The scheduler gives every process a fair turn regardless of what it's doing. No process blocks the scheduler. No stop-the-world garbage collection. P99 latency stays close to P50. For financial platforms where a slow response is a compliance event — a withdrawal that times out, a KYC check that stalls — consistent latency matters more than peak throughput.

P99 ≈ P50 · measured in production
04 — Native distribution

Cluster by default.
No infrastructure tax.

BEAM nodes form clusters automatically via libcluster. Processes communicate across nodes transparently. Phoenix PubSub broadcasts across the cluster without Redis. Horde provides distributed process registry and supervisor — processes migrate between nodes on failure without the user noticing. Horizontal scaling is an operational decision. Not an architectural one made at design time.

libcluster + horde · zero additional infrastructure

02 — The language

Compile-time metaprogramming
that produces real code.

Elixir's macro system operates on the AST at compile time. Not string substitution. Not runtime reflection. Macros produce real, optimized function calls — the code you declare compiles to the code you would have written by hand, structured correctly, never missing a case.

Macro system

Declarations that compile to optimized code

This is how Spark DSL works — and how Foundry's entire governance layer is expressed. A resource declaration that reads like a specification compiles to efficient, type-checked Elixir. The human reads intent. The compiler runs implementation.

Pattern matching

Code that reads like the domain it models

A transfer step reads like a business process. A validation reads like a business rule. The language reduces the distance between what a domain expert describes and what the code says. Code that reads like its domain diverges from intent more slowly.

Tooling ecosystem

One ecosystem. Consistent conventions.

Mix, ExUnit, ExDoc, Dialyzer, Credo. One build tool, one test framework, one documentation system, one type checker, one linter. No build tool archaeology six months into a project. No framework churn. The conventions hold.

03 — The domain layer

One declaration.
Every concern derived from it.

The conventional ORM approach gives you database access. Everything else — authorization, validation, API serialization, audit trail, pagination, filtering — you build yourself. Custom, inconsistent across the codebase, accumulating debt from the first sprint. Ash collapses all of this into a single resource declaration.

lib/my_app/finance/ledger_entry.ex 
defmodule MyApp.Finance.LedgerEntry do
  use Ash.Resource,
    domain: MyApp.Finance,
    data_layer: AshPostgres.DataLayer,
    extensions: [
      AshPaperTrail.Resource,   audit trail
      AshArchival.Resource,     soft delete
      Foundry.Analytics.Extension
    ]

# ── attributes ──────────────────────────────
  attributes do
    attribute :amount, MyApp.Money, allow_nil?: false
    attribute :type, :atom,
      constraints: [one_of: [:credit, :debit]]
    attribute :posted_at, :utc_datetime_usec
  end

# ── relationships ───────────────────────────
  relationships do
    belongs_to :wallet, MyApp.Finance.Wallet
  end

# ── actions ─────────────────────────────────
  actions do
    defaults [:read]
    create :post do
      accept [:amount, :type, :wallet_id]
      change set_attribute(:posted_at, &DateTime.utc_now/0)
    end
  end

# ── policies ────────────────────────────────
  policies do
    policy action(:post) do
      authorize_if actor_attribute_equals(
        :role, :ledger_operator
      )
    end
  end

# ── compliance ──────────────────────────────
  compliance do
    links ["RG-UK-014"]
  end
end

04 — Evidence from production

Teams who moved to Elixir
didn't move back.

These aren't benchmarks run in controlled environments. They're production systems, real traffic, real teams, real consequences. The results are consistent enough to be a pattern.

WhatsApp
Erlang/BEAM · acquired by Facebook 2014
benchmark
messaging infrastructure · erlang on beam

Before the Facebook acquisition, WhatsApp ran on Erlang — the language BEAM was originally designed for. The ratio of engineers to users was extraordinary: a team of fifty engineers maintaining infrastructure for five hundred million users. This wasn't an accident of hiring or tooling. It was a structural consequence of the BEAM runtime's concurrency model. Two million concurrent connections per server meant dramatically less infrastructure than any conventional stack would require at the same scale.

2M concurrent connections per server
The secret of WhatsApp's success is that they never built what Erlang gives them for free. — Observed by multiple engineers post-acquisition

Bleacher Report
Ruby on Rails → Elixir/Phoenix
infrastructure cost
sports media · high-spike traffic · 2017

Running a Rails monolith across 150 servers to handle sports event traffic spikes. The architecture was conventional: aggressive horizontal scaling to compensate for thread-per-request overhead. Major sporting events — Super Bowl, World Cup, NBA Finals — produced traffic spikes that required the entire fleet. After rewriting in Elixir and Phoenix, the same traffic ran from five servers. The BEAM concurrency model handled spike load that required 150 Rails servers because lightweight processes scale concurrency without OS thread overhead.

150→5 servers for equivalent peak traffic
~95% infrastructure cost reduction
The rewrite wasn't planned as a cost-cutting exercise. The cost savings were a consequence of the concurrency model.
Discord
Go → Elixir for read states service
operational simplicity
5M concurrent users · stateful connections

Discord ran their read states service — tracking which messages each user had read — on Cassandra and Go. At scale the Cassandra cluster degraded under pressure in ways requiring constant operational intervention. Fan-out on high-follower servers caused latency spikes that affected the entire service. The rewrite to Elixir handled the same load with dramatically simpler operational requirements. The BEAM concurrency model handled fan-out that required architectural gymnastics in Go. Their engineers noted the Elixir code was smaller, more correct, and handled edge cases the Go service had treated as operational problems.

5M concurrent users — simpler operations
↓ ops manual intervention eliminated
We moved from Go to Elixir for performance. We stayed for operational simplicity we didn't expect to find.
Bet365
Erlang/Elixir · regulated financial transactions
regulated domain
iGaming · financial transactions · real-time odds

One of the world's largest online gambling platforms runs its core platform on Erlang and Elixir. Financial transactions, real-time odds updates, concurrent user sessions during major sporting events — all on BEAM. The regulated domain requirements — consistency, audit trail, fault tolerance, transaction correctness — are exactly the properties BEAM was designed for in 1986. The choice isn't coincidental. When a failed transaction must be recorded rather than silently lost, when consistency matters more than eventual consistency, BEAM's design constraints align with regulated domain requirements more closely than any other mainstream runtime.

millions of financial transactions daily
real-time odds across concurrent sessions
The failure modes BEAM was designed to prevent are precisely the failure modes that matter in a regulated financial domain.
Pinterest
Go → Elixir for notification infrastructure
code simplicity
billions of notifications · backpressure management

Pinterest's notification system was built on Go and handled billions of notifications. The system worked but required careful management of goroutine pools, buffer sizes, and backpressure — high operational surface area for what was fundamentally a message routing problem. The Elixir rewrite used OTP's built-in process model to handle the same concurrency with significantly less operational complexity. The supervision tree handled failures that previously required custom recovery logic. The code was smaller. The operational requirements were lower. The failure modes were fewer.

smaller codebase for equivalent capability
↓ ops custom recovery logic eliminated
OTP gives you the failure recovery logic for free. In Go we wrote it ourselves. We got it wrong more than once.

The honest case for each alternative

JAVA / SPRING BOOT defensible

The largest financial systems in the world run on Java.

Enormous talent pool. Comprehensive Spring ecosystem. JVM tuned over thirty years. Project Loom closes the concurrency gap significantly. GraalVM native compilation eliminates startup time. If you have a Java team and Java vendor relationships, starting here is a reasonable choice.

The gap: domain modeling is four separate layers — persistence, authorization, API, audit. All custom, all inconsistent, all accumulating debt. There is no Ash for Java. The system map Foundry produces requires a single declarative source — Spring annotations scatter domain facts across multiple files.
GO wrong tool

Excellent for infrastructure. Wrong for domain platforms.

Go was designed for networked infrastructure services — DNS servers, proxies, orchestrators. It excels there. Kubernetes, Docker, Terraform are Go. The concurrency model is good. The deployment story is excellent. Single binary, fast compilation, low memory.

The gap: domain modeling in Go is structurally verbose. No generics with expressive power. No pattern matching. No compile-time metaprogramming. Expressing a complex domain rule requires more code than the rule warrants. Discord moved Go code to Elixir specifically because BEAM handled their concurrency problem better. Go is the right choice for infrastructure tooling. It is the wrong choice for complex domain platforms.
RUST wrong tradeoff

Memory safety without GC. Irrelevant for domain logic.

Rust's memory safety guarantees are transformative for systems programming. WebAssembly, embedded systems, browser engines. The type system is the most sophisticated of any mainstream language. The borrow checker prevents entire classes of bugs at compile time.

The tradeoff is wrong for domain platforms. The bugs Rust prevents — use after free, buffer overflows, data races in shared memory — don't happen in GC languages. Domain platforms fail because a business rule was wrong, a state machine transition was missing, a compliance obligation wasn't linked to a test. Rust doesn't prevent those. GC already handles memory. You pay the borrow checker's productivity cost without getting the benefit.
PYTHON / DJANGO scale wall

Fastest to prototype. Expensive to scale.

Python's productivity advantage at early stage is real. Django is mature with thirty years of security patches. The data science ecosystem is Python and if your platform needs deep ML integration, Python interoperability matters. FastAPI is excellent for async services. The talent pool is the largest of any language.

The GIL means threads don't run in parallel. Real concurrency requires multiple processes — multiple memory spaces, IPC overhead. At any meaningful scale, Python needs significant infrastructure investment to compensate for what BEAM provides natively. Django's ORM is persistence only — authorization, API, audit trail are separate constructs. Technical debt accumulates faster because the framework doesn't enforce consistency.
NODE / TYPESCRIPT scale wall

Largest talent pool. Highest long-term maintenance.

JavaScript/TypeScript is the most widely known language in the world. The talent pool is enormous. TypeScript's type system has matured significantly. For teams with strong JavaScript experience and I/O-bound workloads, Node.js is productive and deployable. The async model handles concurrency well for its intended use case.

The single-threaded event loop means CPU-bound domain logic blocks the event loop. Type safety erodes at system boundaries — JSON parsing, external API calls — in ways that require explicit handling. The npm ecosystem's size is also its liability: package selection, version conflicts, supply chain security. There is no Ash equivalent. NestJS adds structure but it's a long way from declaring a resource once and deriving all concerns from it.
KOTLIN / SPRING better Java

Genuinely better than Java for domain modeling.

Data classes eliminate Java's verbosity. Sealed classes enable proper algebraic data types. Extension functions produce more expressive domain code. Coroutines are more ergonomic than Java threading. Arrow adds functional patterns. If someone said "we're building on Kotlin with Spring Boot and Arrow" the technical argument against it is weaker than against Java.

Closest to the right direction among JVM options. The domain modeling is more expressive. The coroutines model closes some of the concurrency gap. Still the same fundamental issue: no single declarative source. The system map remains an approximation. No Ash equivalent exists for Kotlin.

06 — What Foundry adds

The stack removes infrastructure complexity.
Foundry removes domain complexity.

BEAM and Ash give you a runtime and framework where hard technical problems are structurally absent or well-solved. What they don't give you is governance of the domain itself — how complex platforms are built, evolved, and maintained as teams change and requirements shift. Foundry adds three optimization layers on top.

Layer 01 — Compile-time

Domain governance at
the compiler level

The Ash DSL is validated by the compiler. Foundry extends this with SparkLint — Foundry's invariant rules that validate domain correctness beyond syntax. A sensitive resource without paper trail is a lint error. A compliance-linked action without test coverage is a lint error. The compiler prevents syntactic mistakes. Foundry prevents semantic ones.

Layer 02 — Generation

AI that knows your domain
completely

Conventional AI tools generate plausible code from training data. Foundry's copilot receives full project context — stack versions, resource declarations, compliance links, relevant ADRs — before every session. It generates correct Ash code for your exact version, constrained by your invariants, classified by your change model. Generation is faster because there's no ambiguity. Correct because constraints are explicit.

Layer 03 — Preservation

Institutional knowledge as
executable constraints

The most expensive problem in platform engineering is knowledge decay — constraints in engineers' heads, decisions made for reasons nobody remembers. Foundry encodes this as ADRs the linter references, spec-kit documents the copilot reads, and invariant rules that fire at the exact moment a violation is about to be introduced. The knowledge doesn't decay because it's executable.

07 — The structural argument

Why Foundry requires
this stack specifically.

Foundry's core thesis — the codebase is the specification, the system map is a lossless rendering of the domain, the copilot reads live declarations rather than approximating them — is only possible because Ash resources are a single declarative source of truth.

In a Spring Boot application, the domain model is scattered. Annotations on entity classes. Separate service beans. Separate repository interfaces. Separate security configuration. Separate API controllers. A system map of a Spring Boot application is a reconstruction from multiple sources that can and do drift from each other. It's an approximation, not a rendering.

In an Ash application, a resource is the source of truth for all concerns. The system map is a direct rendering of declarations that the compiler validates. It cannot be wrong because it's generated from the same source the compiler uses. That property — lossless, always current, compiler-validated — only exists because of how Ash's DSL is constructed using Elixir's macro system.

Spring Boot

Domain facts scattered across @Entity, @Service, @Repository, @Controller, @PreAuthorize annotations. System map is a reconstruction — approximate by construction. Cannot be made lossless without rewriting how Spring works.

Ash resource

All domain facts in one declaration. The system map reads this declaration directly — the same source the compiler validates. Lossless by construction. Cannot drift because there's nothing to drift from.

If Ash existed for Java, Foundry could target Java. It doesn't. And building something like Ash for Java would be a multi-year project — Java's type system and reflection model don't support compile-time DSL construction the way Elixir's macro system does.

We didn't choose this stack
because Java doesn't work.
We chose it because this works better.

BEAM's concurrency, OTP's fault isolation, Elixir's compile-time metaprogramming, and Ash's declarative domain model combine into a platform where the hardest problems of complex system construction are structurally absent. Foundry builds on that foundation to eliminate the domain complexity that the stack alone doesn't address. The result is platforms that scale without drama, evolve without entropy, and outlast the teams that built them.

Bleacher Report proved the infrastructure economics. Discord proved the operational simplicity. WhatsApp proved the scale ceiling. Bet365 proves it daily in a regulated financial domain. The pattern is consistent enough to be a decision, not a bet.
